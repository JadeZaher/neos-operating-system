# NEOS — New Earth Operating System

## What It Is

NEOS is a modular governance skill stack — a non-sovereign coordination architecture that defines how authority, agreements, decisions, economics, and coordination function across autonomous units without centralized coercive authority.

It is civilizational middleware: the protocol layer between individual autonomy and collective coordination.

**NEOS is NOT**: a nation-state, a DAO, a democracy, a token-governed system, a belief system, or a political ideology. It is an OS layer.

## Target Users

1. **AI Agents** — Claude-ready skills that help individual participants communicate within the system, perform obligations, navigate agreements, request resources, resolve conflicts, and understand their authority scope
2. **Ecosystem Architects** — Human designers who adopt NEOS core as a blueprint and configure it for their own governance ecosystem

## First Ecosystem: OmniOne

The first ecosystem instantiating NEOS is **OmniOne**, stewarded by **Green Earth Vision (GEV)**, a 501(c)(3) non-profit. OmniOne operates out of Bali with plans for Costa Rica, Mexico, Brazil. Physical manifestation includes the **SHUR (Synergy Hub Universal Residencies)** network.

OmniOne's organizational structure:
- **Town Hall (TH)** — Any active contributing ecosystem member (post-NEXUS onboarding)
- **Agents of the Ecosystem (AE)** — Those actively building the ecosystem
- **GEV (X)** — Organizational design firm helping create new paradigm organizations
- **OMNI Steward Council (OSC)** — Fully committed stewards
- **Current-Sees** — Influence currencies (everyone gets 111, equal when affected)
- **Profiles**: Co-creator, Builder, Collaborator, TownHall

## The 10 Skill Layers

```
I.    Agreement Layer         — Lifecycle of agreements (create, amend, review, sunset)
II.   Authority & Role Layer  — Scoped authority, role definition, domain boundaries
III.  Decision (ACT) Engine   — Advice → Consent → Test decision protocol
IV.   Economic Coordination   — Resource requests, funding pools, access economy
V.    Inter-Unit Coordination — Cross-AZPO requests, shared resources, federation
VI.   Conflict & Repair       — Mediation, coaching escalation, repair protocols
VII.  Safeguard & Capture     — Detection of capital/charisma/emergency capture
VIII. Emergency Handling       — Temporary authority expansion with auto-revert
IX.   Memory & Traceability   — Decision logs, agreement versioning, precedent search
X.    Exit & Portability      — Voluntary exit, data portability, graceful degradation
```

## Architecture: Core with Examples

`neos-core/` contains the abstract NEOS protocol with OmniOne as inline examples and defaults throughout. This means:
- Every skill is NEOS-generic at its structural level
- OmniOne-specific details appear as clearly marked examples, defaults, or configuration blocks
- Another ecosystem can fork and replace the example blocks with their own configuration

## Core Structural Principles (Non-Negotiable)

1. **No Sovereign Authority** — No permanent centralized override
2. **Scoped Authority** — Authority exists only within defined domains, must expire or be reviewable
3. **Voluntary Participation** — Exit always structurally possible
4. **Capital ≠ Power** — Economic contribution does not grant governance authority
5. **Solutionary Decision-Making (ACT)** — Advice → Consent → Test
6. **Local Failure Containment** — Autonomous units, failure doesn't cascade
7. **Explicit Agreements** — Written, scannable, traceable, revisable
8. **Capture Resistance** — Resists capital, charismatic, emergency, informal capture
9. **Pluralism** — Multiple value systems coexist via agreement, not belief
10. **Emergency Contraction** — Temporary expansion auto-expires

## Success Criteria

**Primary**: OmniOne members can use the skills daily for real governance — making decisions, forming agreements, requesting resources, resolving conflicts, onboarding members, and coordinating across AZPOs.

**Secondary**: The skill stack is structured so another ecosystem could adopt NEOS core and write their own configuration overlay.

## Known Failure Modes to Guard Against

- Token-weighted governance drift
- Informal leadership capture
- Voter fatigue collapse
- Emergency power permanence
- Cross-unit fragmentation
- Capital influence creep
- Decision bottleneck centralization
- Moral enforcement replacing structural clarity
- Hidden authority layers
- Exit becoming economically punitive
